# 创作模块出图质量优化：诊断结论与 A/B 实验体系设计

> 日期：2026-07-05
> 状态：设计已与用户对齐，待启动实施
> 讨论范围：策略与方法论（本文档为 brainstorming 讨论的持久化产物，实施计划另行编写）

## 1. 背景与问题

生产环境近期出图效果不达预期，三大高频问题（按严重度排序）：

1. **角色脸部崩坏（最严重）**：脸部外观与角色本人严重不符
2. **角色视觉锚点丢失**：头饰/发饰等识别性特征在出图中频繁丢失
3. **腿部/脚部/袜子细节崩坏**：大腿过粗、比例失调、脚部夸张、袜子违和（腿脚袜是业务核心卖点，见 memory「创作模块腿脚展示第一原则」）

诊断素材：`docs/creation_pipeline.md`（五节点业务流）、`prompt_case1.txt`（Castorice，鸭子坐）、`prompt_case2.txt`（Hysilens，高背椅吹泡泡）两份生产最终 Prompt、节点 1–4 全部 Prompt 源文件。

## 2. 诊断结论

### 2.1 脸部崩坏：三个元凶

| # | 元凶 | 证据 |
|---|------|------|
| 1 | **表情复杂度超出模型能力**（用户猜想，成立） | case1 神态段同时要求"害羞+局促+极致信任+含情直视+浅笑+歪头"6 个微表情信号；文生图模型表情可控粒度仅"微笑/平静/闭眼"级别 |
| 2 | **Prompt 内部自我矛盾**（锁脸 vs 改脸） | 固定段要求"与参考图完全一致，不要改脸"，神态段却要求参考图中不存在的复杂表情，锚点段直接改写眼型（case1"微微下垂（无辜感）的高饱和度亮紫瞳"） |
| 3 | **脸部像素预算过小** | 全身构图（主体占比 55–65%）+16:9 下脸部区域极小，却承载最密集的要求；材质段的 SSS/物理折射等写实渲染词汇与 3D toon 风格段互相拉扯 |

### 2.2 锚点丢失：锚点写法问题

- **数量超标**：模板要求 3–5 个，实际每条锚点内嵌 3–6 个子特征，真实锚点数 15+，注意力摊薄后随机丢弃
- **文学化标签**："神性的冠冕"、"不对称法则"、"灵魂眼瞳"——标签本身是噪声，视觉名词被埋在修辞里
- **与参考图脱钩**：锚点为纯文本，无一句绑定回 5 张参考图，文本与图像成为两个独立信号源

### 2.3 腿脚袜崩坏：Prompt 问题 + 姿势可行性各占一半

- **姿势超纲**：case1 鸭子坐（W-sitting）是图像模型公认高难姿势；case2 为双平面腿部布局。节点 2 有"严禁复杂多肢交叉"约束但仅在种子层生效，step1 展开姿势细节时无难度门控
- **解剖学描述误导**："足弓线条""骨骼块面"等词诱发过度解剖细节
- **正负 Prompt 直接矛盾**：正文要"丝滑高光"/Negative 禁"反光丝袜质感"；正文要"朦胧透出皮肤"/Negative 禁"露出脚趾"

### 2.4 结构性问题（用户未列出、诊断新发现）

1. **流水线元数据泄漏**：`[COMPOSITION_DECISION]` 决策块与 `[SHOOTING_ANGLE] three_quarter` 枚举码被原样发给图像模型
2. **step2 职责导致复杂度只增不减**：step2 任务是"补充高级配置"，good_template 范例又长又华丽，Prompt 逐轮膨胀
3. **节点 4 筛选目标错位**：review 按"文本质量"挑选，系统性奖励华丽 Prompt——恰是最易崩坏的那类；全流程无"图片实际效果"回流闭环
4. **Negative Prompt 机制误用**：Nano Banana（Gemini 系）无真正负通道；`(shoes:1.3)` SD 权重语法无效；负面名词罗列可能反向注入概念

### 2.5 现有单图审核节点（prompt_review.py）失效

多肢等严重崩坏漏检率高，根因是"一次调用问十件事"。该节点推倒重设计（见 §5）。

## 3. 总体路线：漏斗式两阶段（路线 C）

> 备选路线 A（外科手术式逐项修复，归因清晰但周期长、不治架构病）与路线 B（最终 Prompt 架构重写，收益大但一轮改几十个变量无法归因）被否决，采用两者的漏斗组合。

- **第一阶段（全局假设检验）**：单一大变量 A/B——"6 规则瘦身版 vs 生产原版"，验证"复杂度过载是主因"这一总假设
- **第二阶段（按结果分流）**：
  - 瘦身显著改善 → 架构没问题，转入局部专项精细 A/B（脸部 → 腿脚，按严重度排序）
  - 改善有限 → 问题在架构层，转入模板重写，且已有第一阶段数据辅助定位
- **配套基础设施先行**：实验运行器（§4）+ 新核对节点（§5），两者先于 exp001 落地

### 已确认的实验优先级

1. Prompt 瘦身（全局，唯一可能同时改善三大问题的单变量）
2. 脸部专项（表情白名单深化 + 锁脸矛盾消除）
3. 腿脚专项（姿势难度门控 + 矛盾清理）；锚点问题预计瘦身已部分解决，余量靠核对节点量化后再补

## 4. 实验运行器（experiments/）

**形态**：项目根新增 `experiments/` 目录，纯后端命令行工具，与 `app/` 并列。绕过前端与 DB 任务流，复用现有 service 函数（step1/step2 装配、Nano Banana 调用）。

```
experiments/
├── runner.py            # 入口：python -m experiments.runner --config configs/<exp>.yaml
├── configs/             # 实验配置（yaml，进 git = 实验可复现）
├── fixtures/            # 基准集：种子清单 + 角色映射 + 锚点核对清单（进 git）
├── variants/            # Prompt 变体（第一轮为人工改写的静态文件，进 git）
├── checker/             # 新核对节点（见 §5）
├── report.py            # 汇总统计 + 生成人工盲评 HTML
└── results/             # 运行产物（gitignore）
    └── exp001/
        ├── prompts/     # 实际发给 Nano Banana 的最终 Prompt 全文存档
        ├── images/      # 按 变体/种子/序号 组织
        ├── checks/      # checker 逐图 JSON 结论
        └── review.html  # 人工盲评对照页
```

**关键设计决策**：

1. **运行环境**：实验在生产环境运行（本地硬件受限），代码与配置 git 提交 → 生产同步 → 运行 → `results/expNNN/` 整目录打包拷回本地 → 本地跑 report.py 出统计与盲评页
2. **并发硬约束**：并发上限 ≤ 10（用户指定），使用独立信号量（不复用 `task_concurrency`，避免与生产任务叠加突破上限），配置可调、默认 10
3. **变体定义**：变体是"生成配方"；但第一轮瘦身版例外——**由 Claude 离线人工改写**基线 Prompt（逐份应用 6 规则），静态文件进 variants/。理由：先验证规则本身有效性；LLM 自动改写会引入遵从性新变量污染第一轮实验。自动化是规则验证通过后的第二步
4. **同种子同基线**：对照组与实验组共用同一批 step1/step2 基线 Prompt（对照组用原文、实验组用改写版），唯一差异是改写本身；基线生成一次、冻结存档进 git
5. **人工盲评页**：review.html 纯静态单文件、图片相对路径；每图三组评分（脸部：正常/轻微异常/崩坏；锚点：齐全/部分丢失/严重丢失；腿脚袜：正常/轻微异常/崩坏）+ 自由备注；**图片乱序且不显示所属变体**（盲评防偏见）；导出 JSON 供 report.py 统计

## 5. 新核对节点（checker/）

**核心原则**：一次只问一个问题、客观问题优先、生成图与参考图直接对照。替代现有 `prompt_review.py`。

### 5.1 第一层：结构完整性检查（全自动）

检查项**按生产真实崩坏分布定义**（用户提供）：高频崩坏为多腿、颈/腰异常扭曲、两个上半身（躯干重复）、背景家具结构异常。（五官扭曲、双脚出画几乎不发生，不设检查项。）

| 检查项 | 问题形式 | 输出 |
|---|---|---|
| 多腿 | 「图中人物有几条腿？只回答数字」 | 计数（真值 2） |
| 躯干重复 | 「是否出现一个以上的上半身/躯干，或躯干与头部数量不匹配？」 | 是/否 + 一句话理由 |
| 颈/腰扭曲 | 「颈部或腰部是否存在违反人体结构的异常扭曲、拉伸或错位？」 | 是/否 + 一句话理由 |
| 背景家具崩坏 | 「家具是否存在结构性错误（椅腿数量异常/结构穿插/透视崩塌）？」 | 是/否 + 一句话理由 |

- 每项独立一次视觉 LLM 调用，只喂生成图（不喂参考图，避免干扰计数）
- **"是非 + 一句话理由"机制**（结构合理性类检查项）：强制模型描述其所见结构，提升判断真实性；校准阶段可通过理由甄别真检出 vs 蒙对

### 5.2 第二层：锚点核对（全自动）

1. **锚点清单预生成**（每角色一次）：`anchor_extractor.py` 从 `chara_profile_final.md` + 参考图提取可判定视觉事实清单（如 Castorice：「头戴黑荆棘粉白花冠」「额前水滴形深紫宝石」「精灵尖耳」），**清单须用户人工审定后固定进 fixtures**
2. **逐项核对**：每锚点一次调用，输入 = 生成图 + 1 张最相关参考图（头饰用 face_close.png），问题形如「对比两图：第一张人物是否佩戴与第二张相同的花冠？是/否/无法判断」

产出：每图 JSON（锚点命中 + 结构结论），汇总自动计算**锚点保留率、多腿率、躯干重复率、颈腰扭曲率、背景崩坏率**。

### 5.3 主观项分工（方案 C，已确认）

脸崩程度、腿脚审美、整体氛围等主观项**不进 checker**，由用户在 review.html 人工盲评；checker 客观指标与人工评分在 report.py 合流为一份实验报告。

### 5.4 校准前置（上线门槛)

checker 使用前先跑校准集：生产历史出图中已知崩坏/正常各 10–20 张（用户挑选并标注真值），验证各检查项准确率；不达标项（如多肢检出率 < 90%）降级为人工评判。

### 5.5 成本估算

每图约 8–12 次视觉 LLM 调用；36 图一轮约 400 次调用内，并发 ≤ 10，走 `yibu_gemini_infer` 多模态通道，预算已确认充足。

## 6. Prompt 瘦身规则（6 条）

第一阶段实验的核心变量。第一轮由 Claude 人工改写基线 Prompt，产物经用户过目后进 variants/。

> **红线（用户确认的最佳实践，不可违反）**：结构化字段骨架原样保留（任务目标/锁定身份/视觉锚点/统一风格/背景场景/姿势/服装/脚部袜子/神态/镜头构图/硬约束/光影/材质/Negative，一个不删不合并）；专业构图/布光词汇全部保留（三点布光、Key/Fill/Rim、等效焦段、Cinematic DOF、God Rays、bloom 等）。瘦身砍的是"糖分"，不是骨架。字数预期从 1500–2000 字压至约 800–1000 字。

| # | 规则 | 要点 |
|---|------|------|
| 1 | **表情降级白名单** | 神态段=单一主表情+视线方向；白名单：平静/柔和微笑/闭眼/微微含笑/专注；禁止多微表情叠加 |
| 2 | **消除锁脸 vs 改脸矛盾** | 删除一切改写参考图五官的描述；脸完全交给参考图；锚点只保留非五官视觉物件；神态段不描述眼睛长相 |
| 3 | **锚点收敛并绑定参考图** | 3–5 个纯视觉名词、每项 ≤8 字、剥离文学标签（"神性的冠冕"→"黑荆棘花冠"）；每锚点加"（见参考图）"；一条锚点只含一个物件 |
| 4 | **删文学修辞与解剖学描述** | 删除对象精确限定为：文学抒情（"如温玉般"）、设计意图旁白（"传达卸下心防"）、解剖学词汇（"足弓线条""骨骼块面"）；专业摄影术语不属于删除范围；腿脚袜段只留可视化客观描述（款式/颜色/透明度/长度/贴合度） |
| 5 | **清除正负矛盾** | 逐条比对正文与 Negative，删互相打架项；原则：正文说要什么，Negative 只放正文未提及的高危项 |
| 6 | **剥离机器码 + Negative 改写** | 删 `[COMPOSITION_DECISION]` 块与枚举码行（构图信息改自然语言保留在镜头段）；删 SD 权重语法；"不穿鞋"类改正文正向表述；Negative 只留 3–5 条最高危名词短语 |

**打包策略（已确认）**：第一阶段 6 条规则打包为一个"瘦身版"整体 A/B；若结果为部分改善或某指标恶化，再对可疑规则做针对性消融——"先打包、按需消融"。

## 7. exp001 实验设计

### 基准集 benchmark_v1

- **角色**：Castorice（`mchar_3695c70ca7`）、Hysilens（`mchar_50c51e6e37`）——两者即 case1/case2 来源，天然回归样本；资料完整性已核实（chara_profile_final.md + 5 张标准参考图 slot）
- **种子**：每角色 3 条共 6 条，覆盖姿势难度梯度：每角色 1 简单（普通坐/倚靠）+ 1 中等（盘腿/侧坐）+ 1 高危回归（Castorice=case1 鸭子坐种子；Hysilens=case2 高背椅吹泡泡种子）；从既有 seed_draft.json 挑选 + case 还原，Claude 起草、用户审定
- **基线 Prompt**：每种子走现有 step1→step2 生成一次（case1/case2 可直接用生产原文），冻结存档进 git

### 对照设计

- A 组（对照）=基线原文；B 组（实验）=同一 Prompt 的 6 规则瘦身版
- 每种子每组 **3 张**（用户指定，加快验证）→ 6×2×3 = **36 张图**
- 同 aspect_ratio、同参考图输入，唯一变量为 Prompt 文本
- 样本量说明：单格 3 张噪声较大，结论只看 6 种子汇总后的组间差异，不看单种子

### 预注册结论标准（防事后解读偏差）

| 结果 | 判定 |
|---|---|
| 任一核心指标（脸部崩坏率/腿脚崩坏率/锚点保留率）改善 ≥ 20pp | 显著改善 → 采纳瘦身、进入第二阶段专项 |
| 主要指标全部改善 < 10pp | 改善有限 → 转入架构重写路线 |
| 任何指标恶化 ≥ 15pp | 触发对应规则的消融排查 |
| 中间情况 | 用户结合看图主观判断 |

### 执行顺序

1. 搭 `experiments/` 骨架 + checker → git 提交
2. checker 校准（历史真值图 10–20 张/类，用户挑选标注）
3. 基线 Prompt 生成冻结 + Claude 人工改写瘦身版 → 用户过目 → git 提交
4. 生产环境跑 exp001（并发 ≤ 10），results 打包拷回本地
5. 本地 report + 用户盲评 → 合流结论 → 决定第二阶段路线

## 8. 已确认的关键决策记录

| 决策点 | 结论 |
|---|---|
| 验证方式 | 小批量单变量 A/B 对照实验；预算充足；授权为实验改造业务流程 |
| 评判分工 | 方案 C：客观项（锚点/结构）LLM 自动核对，主观项（脸/腿脚审美）用户人工盲评 |
| 现有 review 节点 | 效果差（多肢漏检），推倒重设计为逐项独立核对 |
| 实验优先级 | 瘦身 → 脸部专项 → 腿脚专项 |
| 实验载体 | 方案 A 轻量脚本层；实验期不动前端；结论稳定后再考虑升级进正式业务流 |
| 运行环境 | 生产环境运行（git 同步），结果拷回本地评判 |
| 并发上限 | ≤ 10（独立信号量） |
| 检查项定义 | 按真实崩坏分布：多腿/躯干重复/颈腰扭曲/背景家具崩坏；"是非+理由"机制 |
| 瘦身红线 | 结构化字段骨架 + 专业构图/布光词汇原样保留 |
| 批量 | 每种子每组 3 张，共 36 张 |
| 瘦身验证方式 | 6 规则打包 A/B，按需消融；第一轮人工改写不走 LLM 自动化 |
