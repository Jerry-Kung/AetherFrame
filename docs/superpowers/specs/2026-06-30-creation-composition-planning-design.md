# 创作模块构图规划优化 · 设计文档

> 状态:设计待用户审阅
> 范围:仅顶层策略 / 方法论层面,不涉及代码级实现细节
> 起源:针对创作模块两个系统性问题(长宽比与内容失配 / 姿势镜头千篇一律)的工作流升级

---

## 1. 背景与问题

经实测,AetherFrame 创作模块当前在角色服饰、室内布景、光影粒子等细节维度已经达到令人满意的水准,但在顶层构图层面有两个显著系统性问题:

**问题 1 — 长宽比与图片内容适配度不佳**

当前长宽比在创作任务启动前由用户在入口处指定,贯穿整个创作链路(场景生成 / 姿势规划 / Prompt 编写 / 出图)始终不变。以最常用的 16:9 为例:横向卧姿契合,但居中坐姿就被迫用大面积背景填充画面两侧,导致角色画面占比过低。由于图片在外部平台首先以缩略图形式呈现,缩略图中角色占比不足直接削弱视觉吸引力。

**问题 2 — 角色姿势 / 镜头视角千篇一律**

尽管创意方向、种子提示词、预生成 Prompt 各阶段都已引入多样化机制,实测产出图仍高度集中于侧卧 / 躺卧 / 抱膝坐等少数姿势,且镜头视角雷同。在缩略图列表中,视觉效果接近"同一个人换不同背景装饰"。根因在于工作流缺少专门的"构图规划"环节,把整个构图决策完全交给 LLM 自由发挥,LLM 在概率最大化倾向下会反复收敛到训练数据中的高频构图。

---

## 2. 不可让步的两条原则

### 原则 1:腿脚自然展示是第一硬约束

自然展示角色腿部与脚部(穿袜不穿鞋)是当前产出图的**核心竞争力**,也是粉丝吸引力的最关键因素。

- 任何构图字段、枚举档位、镜头组合,设计前都必须先过一遍"会不会遮挡 / 裁掉腿脚"过滤。
- 严禁出现具有 fetish 倾向的镜头语言:脚部特写、袜子特写、袜口装饰特写、纯下半身镜头等一律不得作为枚举档位。
- 即便为了"提升多样性""提升缩略图吸引力",也不得以削弱腿脚展示为代价。可以牺牲一定的构图多样性,但腿脚展示不可让步。

### 原则 2:LLM 实现能力是设计上限

当前文生图 LLM 在多关节、复杂腿脚交叉、悬空、扭曲等复杂姿态下容易翻车(多肢错误、关节反向、肢体粘连)。

- 不设计复杂的腿脚动作枚举,腿脚摆放用自然语言 1–2 句简洁描述。
- 用更稳健的"机位组合多样性"(同姿势不同视角)替代"姿势细节多样性",避开 LLM 翻车区。

---

## 3. 核心策略

**不新增工作流节点**。把构图规划字段集**分散嵌入现有 4 个节点**(创意方向 / 种子提示词 / 预生成 step1 / 预生成 review),每个字段按其最自然的语义归属落位。

**分 6 个独立可回滚切片(S0–S5)** 逐步推进,每切片只动 1–2 个文件,可独立上线、独立验证、独立回退。

**所有枚举值**(姿态家族 / 机位方位 / 机位高度 / 视线 / 主体占比下限 / 长宽比集合等)**必须支持持续迭代更新**,不得在 Prompt 模板中硬编码自然语言形态。详见 §7。

---

## 4. 字段 → 节点 分配方案

| 字段 | 落入节点 | 表达形态 |
|---|---|---|
| `aspect_ratio` 长宽比 | 入口 + step1 | Auto 模式 step1 决策(9 档全集) / 手动模式入口指定(5 档保留) |
| `subject_area_min` 主体占比下限 | step1 | 4 档,初始值:0.45 / 0.55 / 0.65 / 0.75 |
| `home_setting` 居家背景框架 | **创意方向** | 自然语言短语,如「卧室大床」「客厅沙发」「日式榻榻米」「飘窗台」「书房地毯」 |
| `pose_family` 姿态家族 | **种子提示词** | 6 档,初始值:坐姿 / 躺姿 / 跪姿 / 蹲姿 / 倚靠 / 盘腿坐 |
| `pose_detail` 腿脚摆放细节 | **种子提示词** | 自然语言 1–2 句,作为种子文本的一部分 |
| `shooting_angle` 机位方位 | step1 | 5 档,初始值:正面 / 3/4 正面 / 侧面 / 3/4 背面 / 背面(回眸) |
| `camera_height` 机位高度 | step1 | 4 档,初始值:略仰 / 平视 / 略俯 / 大俯 |
| `gaze_direction` 视线 | step1 | 5 档,初始值:看镜头 / 3/4 看出画 / 侧面看 / 看下方 / 看远处 |
| 反同质化挑选 | review | 维度差异分 + 权重表预留 |

**所有枚举值标注"初始值"** — 强调档位会在使用过程中持续迭代,不是固定不变的设计常量。

## 5. `home_setting` 与姿势的联动机制

### 由创意方向产出

创意方向的 7 小节里「推荐场景」字段就是描述空间气质的最佳载体。在该字段下方新增一行结构化输出:`home_settings: [短语1, 短语2, 短语3]`,1–3 个具体的居家背景框架短语。

### 数据流转

`description` markdown 保持不变;`home_settings` 作为方向行的新字段持久化。通过现有"创意方向折叠进 `{seed_prompt}`"的路径(见 `claude_docs/feature_creative_direction/design.md` §7.3.1)自然向下传递 — 种子生成 Prompt 已能拿到方向全文,不需要新增传递通道。

### 与姿态家族的软联动

种子生成 Prompt 中加入软约束:每条种子选用方向提供的 1–3 个背景之一,并选择与该背景天然契合的姿态家族;8–10 条种子在 1–3 个背景间均衡分布。

提供一份**软指引参考表**(LLM 可在表外创造性发挥,不是硬映射):

| 背景框架(示意) | 契合姿态家族(示意) |
|---|---|
| 卧室大床 | 躺姿 / 倚靠 / 盘腿坐 |
| 客厅沙发 | 倚靠 / 坐姿 / 躺姿 |
| 日式榻榻米 | 跪姿 / 盘腿坐 / 坐姿 |
| 飘窗台 | 坐姿 / 倚靠 |
| 书房地毯 | 盘腿坐 / 蹲姿 / 跪姿 |

**这份参考表也不得在 Prompt 中硬编码** — 见 §7 的统一原则。

## 6. 六个切片(执行顺序)

### S0 — 创意方向加 `home_settings` 字段

**目标**:为下游姿势规划提供具体的背景框架锚点。

**改动点**:
- 创意方向 Prompt(`creative_direction_prompt`)的 7 小节中,在「推荐场景」小节下方新增结构化输出 `home_settings`(1–3 个短语)。
- 方向行新增字段持久化 `home_settings`。

**独立验证**:重新生成现有方向,人肉审阅 `home_settings` 短语是否合理可用;不影响出图链路。

**回滚成本**:极低。

---

### S1 — Auto 长宽比 + 主体占比下限

**目标**:解决问题 1(长宽比与内容失配 / 缩略图角色被淹没)。

**改动点**:
- 入口长宽比选项:新增 `auto` 选项;手动 5 档保留不变。
- step1 Prompt:Auto 模式下要求 LLM 先决策 `aspect_ratio`(9 档全集)与 `subject_area_min`(4 档),写入模板对应字段。
- Auto 模式安全引导:Prompt 中提示"优先在主流 5 档(`9:16, 3:4, 1:1, 4:3, 16:9`)中选择,除非场景明显适配特殊比例(如 `5:4` 适合接近正方形的居中坐姿、`2:3` 适合竖向全身),才使用扩展 4 档(`4:5, 5:4, 2:3, 3:2`)"。
- 模板(`prompt_template.py`):
  - 移除写死的 `16:9` 硬编码,改为占位符接受 step1 决策结果回填。
  - 「构图硬约束」段新增"角色主体在画面中占比不低于 X%"(X 由 `subject_area_min` 注入)。

**独立验证**:同一种子在 Auto 与 16:9 模式各跑一批,缩略图角色占比目测对比;手动模式行为不变(回归)。

**回滚成本**:极低 — 模板恢复硬编码 16:9 即可。

---

### S2 — 种子提示词加姿态家族均衡 + pose_detail 简洁性约束 + 背景联动

**目标**:从源头打散姿态同质化,并实现 `home_setting` 与 `pose_family` 的联动。

**改动点**:在种子生成 Prompt(`creation_direction_seed_prompt`)中加入以下硬约束:
- **姿态家族均衡**(决策 D1 选项 1):8–10 条种子必须覆盖 ≥4 类姿态家族,且任一类别不超过 3 条。
- **腿脚摆放简洁性**:每条种子必须包含一句简洁的腿脚自然摆放描述(如"一条腿自然搭在沙发靠背上""盘腿坐,脚踝交叠");严禁复杂多肢交叉、悬空、扭曲的姿态描述。
- **反 fetish**:腿脚摆放描述不得带有暗示性或刻意聚焦,以自然居家状态为基调。
- **背景联动**:每条种子选用方向提供的 `home_settings` 之一,并选择与该背景天然契合的姿态家族(参考软指引表,但可创造性发挥);8–10 条种子在 1–3 个背景间均衡分布。

**独立验证**:让现有几个角色重新生成一批种子,统计姿态家族分布 + 背景分布 + 人肉审阅腿脚描述质量。

**回滚成本**:极低 — 旧种子可继续使用,只回滚 Prompt 文件。

---

### S3 — step1 引入机位方位 + 机位高度枚举决策

**目标**:解决问题 2 中的"镜头视角千篇一律"(同姿势出多样性的最强杠杆)。

**改动点**:
- step1 Prompt 新增 **任务步骤 0:先做构图决策**(决策 D2 选项 1) — 在脑补画面之前,先从 `shooting_angle` 5 档和 `camera_height` 4 档中各选一项,后续场景 / 姿势 / 服装 / 神态都围绕该决策展开。
- step1 必须在输出中**显式说明本次选择的机位方位 / 高度组合,以及为什么这个组合与本场景姿势契合**(让决策可追溯,也便于 review 阶段评估)。
- 模板「镜头与构图」段顶部新增 2 行回填位,接受枚举结果。
- **结构化持久化**:step1 决策的所有枚举字段值统一存到任务的 cards JSON 中,为后续可学习机制提供历史数据基础。

**独立验证**:同种子跑 N 张,统计 `shooting_angle × camera_height` 分布是否均衡;盲选对比"S3 前 vs 后"的镜头多样性。

**回滚成本**:中 — step1 输出结构变动,但向后兼容容易(去掉 step1 中的两行枚举要求即可)。

---

### S4 — step1 引入视线枚举决策

**目标**:补齐镜头层最后一个维度。

**改动点**:
- step1 Prompt 在任务步骤 0 中追加 `gaze_direction` 5 档枚举决策。
- 模板「镜头与构图」段顶部回填位再加 1 行。
- `gaze_direction` 值也并入结构化持久化。

**独立验证**:同上,统计视线分布。

**回滚成本**:中 — 同 S3。

---

### S5 — review 加入反同质化评分 + 权重机制预留

**目标**:在质量挑选基础上加多样性挑选,作为兜底;同时为未来的权重学习机制预留装配点。

**改动点**:
- review Prompt(`prompt_review`)新增评分维度:"在质量相近的候选中,优先挑选 `shooting_angle / camera_height / gaze_direction / pose_family` 维度组合差异更大的 N 条"。
- review Prompt 中**独立**预留「维度权重表」段落(本轮渲染为空,默认权重 1.0)。该段与上面的"维度差异分"评分维度逻辑独立:差异分始终生效,权重表在未来启用后用于"质量与差异度都接近时,按权重选偏好组合"的二级排序。两段在 Prompt 中明确分隔,避免装配层混淆。

**独立验证**:对同一批 2N 候选,新旧 review 各跑一次,对比挑出的 N 条构图维度差异度。

**回滚成本**:极低 — 只回滚 review Prompt 文件。

## 7. 枚举值集中管理与持续迭代

**这是顶层原则,不是预留项**。所有枚举值(`pose_family` / `shooting_angle` / `camera_height` / `gaze_direction` / `subject_area_min` / `aspect_ratio` 候选集 / `home_setting → pose_family` 软指引表)在使用过程中将持续迭代优化,设计必须为此提供原生支持。

### 本轮(代码常量阶段)

- 所有枚举值集中定义在**一个 Python 模块**:`app/services/creation_service/composition_dimensions.py`(或等价位置)。
- 模块对外暴露统一接口:`get_dimension_values(dimension_code: str) → List[DimensionValue]`,其中 `DimensionValue` 至少包含 `code`(英文标识)、`display_name`(中文展示)、`description`(中文说明)三个字段。
- 任何节点的 Prompt 装配层(种子生成、step1、review)在需要枚举集时,**必须通过该接口读取并实时渲染到 Prompt 字符串中**,严禁直接在 Prompt 模板里写死任何枚举值的自然语言形态。
- 同理,`home_setting → pose_family` 软指引表、Auto 模式的 9 档候选与主流 5 档子集,也以同样方式集中管理。

**迭代效果**:调整任一档位(增加 / 删除 / 改名 / 改描述) = 改一处常量;不需要动 Prompt 模板,不需要动节点拓扑,不需要 PR 涉及多文件。

### 未来(DB 字典表阶段)

- 同一函数 `get_dimension_values()` 的实现切换到读取 `composition_dimension_values` 表,接口签名不变。
- Prompt 模板完全不动。
- 切换路径见 §8。

## 8. 可扩展 / 可学习机制(后续版本优先实现)

本轮(S0–S5)只动 Prompt 与模板,不建表、不实现学习逻辑。但本节给出**明确的近期路线图**,确保后续实现时不需要大范围调整本轮设计。

### 8.1 学习路径全景

```
        ┌─── 正向学习 ─────────────────────────────┐
        │  上传/标记参考图 → LLM 抽取构图标签 →     │
        │  入库为高权重 combination 模板             │
        │                                           ├─→ 注入构图字段字典表 ─→ Prompt 装配层
        │  种子+生成参数+成图被用户标"赞" →         │  注入权重表(正权重)
        │  入库为加权样本                           │
        └───────────────────────────────────────────┘

        ┌─── 负向学习 ─────────────────────────────┐
        │  用户对成图标"差/封禁"标签 →              │
        │  LLM 抽取该图构图标签+用户原因 →           ├─→ 注入字典表(标 is_active=false / 加风险标签)
        │  入库为负样本                             │  注入权重表(负权重 / 0)
        │                                           │  注入 Negative Prompt 池(供 step1 自动拼接)
        │  用户手工添加"禁止标签"                   │
        └───────────────────────────────────────────┘
```

### 8.2 数据基础设施(后续版本要建)

| 表 | 用途 |
|---|---|
| `composition_dimensions` | 维度字典(`pose_family` / `shooting_angle` / ...);承载 §7 接口 |
| `composition_dimension_values` | 枚举值字典,含 `code`, `display_name`, `description`, `is_active`, `source ∈ {builtin, user, learned_from_ref}`, `risk_tag?` |
| `composition_combinations` | 完整构图配方,可挂参考图、可挂角色 |
| `composition_feedback` | 正/负反馈记录,字段:`combo_or_partial_tags`(JSON), `polarity ∈ {positive, negative}`, `weight_delta`(`float`), `reason_text`, `source ∈ {user_manual, llm_inferred_from_ref, llm_inferred_from_output}`, `applies_to ∈ {global, character_id}`, `created_at` |
| `composition_combo_weights` | 派生表:由 `composition_feedback` 聚合而成的当前生效权重(快照表,避免每次查询都聚合) |
| `composition_negative_prompts` | 负向标签短语池,含 `phrase`, `applies_to_dimension?`, `scope ∈ {global, character, combo}`, `created_at` |

### 8.3 三种学习触发场景的统一接口

为了让后续实现时不被场景细节绑死,所有学习触发收拢到一个抽象函数:

```
record_composition_feedback(
    polarity: positive | negative,
    source: user_manual | llm_inferred_from_ref | llm_inferred_from_output,
    combo_or_partial_tags: dict,   # 完整 combo 或部分维度(允许部分维度学习)
    applies_to: global | character_id,
    reason_text: str,
    weight_delta: float = ±1.0     # 默认正向 +1 / 负向 -1
) → feedback_id
```

四种场景统一通过此接口写入 `composition_feedback`,聚合器异步更新 `composition_combo_weights`:

1. **正向 — 上传参考图**:用户上传 → LLM 抽取构图标签 → `polarity=positive, source=llm_inferred_from_ref`。
2. **正向 — 标"赞"成图**:用户对成图点赞 → 该图的 step1 决策结果已知(本轮 S3/S4 已结构化持久化到 cards) → `polarity=positive, source=llm_inferred_from_output`。
3. **负向 — 标"差/封禁"成图**:用户对成图标差 → 已知其构图标签 → `polarity=negative`;同时可选地让 LLM 分析该图问题并产出一条负向短语写入 `composition_negative_prompts`。
4. **负向 — 手工添加禁止项**:用户在设置面板手工添加禁止标签或禁止组合 → `polarity=negative, weight_delta=-∞`(等价禁止)。

### 8.4 负向反馈的双档制(决策 D3 选项 1)

负向反馈分两档,语义不同处理不同:

- **严格禁止档**(`weight = 0` 或 `weight_delta=-∞`):用于 fetish / 审核风险类。直接从 Prompt 装配的枚举集中剔除,LLM 完全看不到该选项,且追加进 Negative Prompt 池强化拦截。
- **降权档**(`weight ∈ (0, 1)`):用于"效果差但不致命"。仍保留在枚举集中,但通过权重表渲染的"分布偏好"段引导 LLM 减少选取频率。

### 8.5 权重 / 禁止项作用到现有 Prompt 的路径

通过本轮 S0–S5 已经预留的 Prompt 段落自动注入:

| 预留段落 | 装配时填入 |
|---|---|
| 种子生成 Prompt 的「姿态家族分布偏好」段 | 从 `composition_combo_weights` 拉取与 `pose_family` 相关的权重,渲染成"本轮种子应减少坐姿(权重 0.5),增加跪姿(权重 1.5)"等自然语言 |
| step1 Prompt 的「镜头组合分布偏好」段 | 同上,作用于镜头维度 |
| step1 Prompt 的 Negative Prompt 字段填写要求 | 从 `composition_negative_prompts` 拉取适用于当前角色 / 全局的负向短语,要求 LLM 把这些短语合并入 Negative Prompt 段 |
| review Prompt 的「维度权重表」段 | 同分布偏好段,但供 review 阶段加权评分使用 |

### 8.6 本轮(S0–S5)需要为后续学习机制做的最小预留

| 预留项 | 在哪个切片实施 |
|---|---|
| step1 决策结果结构化持久化(`pose_family`、`shooting_angle`、`camera_height`、`gaze_direction`、`aspect_ratio`、`subject_area_min`、`home_setting`)到任务 cards JSON | S3 / S4 完成时一并落 |
| 创意方向产出的 `home_settings` 持久化到方向行 | S0 |
| 种子生成 Prompt 中预留「姿态家族分布偏好」段(本轮渲染为空字符串) | S2 |
| step1 Prompt 中预留「镜头组合分布偏好」段(本轮渲染为空字符串) | S3 |
| step1 Prompt 的 Negative Prompt 字段加「风险标签合并」指令(本轮风险标签源为空) | S3 / S4 |
| review Prompt 中预留「维度权重表」段(本轮渲染为空) | S5 |
| 所有枚举值统一通过 §7 的 `get_dimension_values()` 接口读取 | S2 / S3 / S4 |

**这套预留确保**:后续从代码常量切换到 DB 字典 + 权重表时,不需要改任何 Prompt 模板、不需要改节点拓扑,只需要替换 `get_dimension_values()` 的底层实现,并把权重 / 负向短语 / 风险标签渲染到已经存在的预留段落里。

## 10. 切片执行顺序与依赖

```
S0(创意方向加 home_settings)
   │
   ▼
S1(Auto 长宽比 + 主体占比) ─── 与 S0 无依赖,可并行,但建议先 S0
   │
   ▼
S2(种子提示词姿态家族均衡 + 背景联动)─── 依赖 S0 的 home_settings 数据
   │
   ▼
S3(step1 机位方位 + 高度)
   │
   ▼
S4(step1 视线)
   │
   ▼
S5(review 反同质化评分 + 权重预留)
```

**为什么 S2 排在 S3/S4 之前**:种子是上游产物。先让上游稳定产出多样化姿态家族,下游 step1 看到的种子才是新分布;否则 S3/S4 的镜头多样化作用于老姿态种子,效果会被稀释。

**每切片的回滚条件**:出图质量在该切片上线后出现 2 个以上角色明显回退、或新增多样性维度未生效,即触发回滚;回滚只动该切片改过的 1–2 个文件。
