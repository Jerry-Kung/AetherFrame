# 第二阶段设计：出图细节深度优化（腿脚侧重）+ Feedback Case 数据闭环

> 日期：2026-07-07
> 前置：`docs/superpowers/specs/2026-07-06-exp001b-conclusion.md`（slim 生产落地完成，regression 采纳为正式版本）
> 相关原则：腿脚自然展示第一原则（记忆 creation-leg-foot-exposure-principle）；slim 六条瘦身规则为基准不动摇（记忆 exp001-slim-adopted）；Feedback Case 数据为核心依据（记忆 feedback-case-data-first）

## 0. 命名澄清

本阶段对外简称"腿脚专项"，实为 **出图细节深度优化（腿脚侧重）**：在不破坏整图构图与电影级视觉美观的前提下，深入优化预生成 Prompt 与出图细节，优化方向侧重腿脚。核心目的：在保证整体美观的前提下，让角色腿脚/袜子部位（产出的核心价值点）自然、精致、美观、符合人体工学，同时压制高频崩坏。

## 1. 总体架构

本阶段交付两样东西，主从关系明确：

1. **Feedback Case 数据闭环（长期基础设施）**：出图 → 预留 feedback 待填模板 → 用户人工填写 → 归档为结构化 Case → 可聚合分析。按用户规则（feedback-case-data-first）建立，未来所有实验与生产回捞数据都走这条管道。
2. **exp002 腿脚规则包 A/B 实验（本阶段主线任务）**：验证"腿脚/袜子描写规范"能否把腿脚崩坏率从 ~22% 基线显著压下来。exp002 的盲评表直接复用闭环的 feedback 模板——评完即归档，不做两遍。

实施顺序：先建闭环（薄，主要是格式规范 + 模板生成 + taxonomy 治理 + 聚合分析），再跑 exp002 作为闭环的首个消费者。

### 手段边界（用户裁定）

- **Prompt 层为主**：step1 姿势/画幅决策 + step2 good_template 腿脚段措辞迭代。
- **链路层为备选**：腿脚特写参考图注入、生成参数调整、失败自动重 roll 等，写入本 spec 第 6 节备选路线，满足触发条件才实施，本阶段不做。
- **治理范围**：腿脚（腿/脚/袜）三类崩坏为主线；画风崩坏（写实 3D 玩偶 / 平面 2D）一并纳入——疑与袜子材质过度物理化描述同源。

## 2. Case 数据规范与大批量分析机制

### 2.1 Case 格式

保持用户现行三段结构（seed_prompt / final_prompt / feed_back）不变，在其上增加结构化头部与标签化崩坏分类，使数据既可人读又可机器聚合：

```
Case_exp002_0042:
[meta]
date: 2026-07-07
source: exp002            # exp002 | production | ...
character: castorice
seed_id: cas_hard_wsit
difficulty: hard          # easy | medium | hard
variant: rulepack         # control | rulepack（生产回捞可空）
images: 3
bad: 2
tags: [袜子/上色感, 袜子/皱褶夸张]   # 取自 taxonomy，父/子 形式；可空待填
taxonomy_version: v1      # 打标签时所用 taxonomy 版本，冻结不回改
[seed_prompt]
<该样例种子提示词>
[final_prompt]
<该样例出图 Prompt 全文（含 [COMPOSITION_DECISION] 决策块，存档保留原样）>
[feed_back]
<自由文本，用户手工填写；模板预生成为空位占位>
```

- 图片不落盘（原始图片占空间大，用户明确可不存）；出图产物仍在 `experiments/results/` 临时留存供盲评，盲评完成后只归档 Case 文本。
- 一批一文件，沿用现行 `experiments/cases/xxx_0706.txt` 惯例。
- 归档布局：`experiments/cases/<source>/`（如 `exp002/`、`production/`）。现有 `bad_case/` `good_case/` 保留为历史手工样例。

### 2.2 崩坏分类 taxonomy（受治理的活字典，非冻结表）

taxonomy 是**带版本的字典文件**，随 feedback 积累演化，不是一成不变的固定表。

- **文件**：`experiments/cases/taxonomy.yaml`，含 `version` 与两层结构。
- **两层结构**：
  - 父 tag（大类）：`脚部` / `腿部` / `袜子` / `画风` / `身体比例` / `其他`
  - 子 tag（具体模式），初始集（v1）：
    - 脚部：`简陋`（一条平滑曲线、丧失二次元韵味）、`夸张`（骨骼肌肉棱角/解剖图感）、`比例结构`（过长过大/脚趾变形/缺趾）
    - 腿部：`结构错误`（多肢/扭曲/过粗过细）
    - 袜子：`上色感`（像在脚上涂色/脚趾根根分明/袜口不清）、`塑料袋感`（透明塑料袋套脚）、`皱褶夸张`（过度褶皱/厚度夸张/毛巾感）、`缺失`（袜子完全消失）
    - 画风：`写实化`（写实 3D 玩偶）、`平面2D`（丧失 3D 动漫质感）
    - 身体比例：`整体不协调`
    - 其他：`未分类`
  - tag 书写为 `父/子`（如 `袜子/上色感`），统计可在父层或子层任意聚合。
- **治理机制**：
  1. **新 tag 提取**：每轮归档时，Claude 对 feedback 原文做一次提取——现有子 tag 覆盖不了的问题模式，提议新子 tag，用户确认后写入 taxonomy 并 bump version。
  2. **合并与分层演化**：同义子 tag 合并、过杂子 tag 重新挂父等治理动作随版本演进；每次变更在 taxonomy 文件内记录 `旧tag → 新tag` 映射表（`aliases`）。
  3. **历史数据不改写**：Case 原文里的 tags 与 `taxonomy_version` 永不回改；聚合分析时按 aliases 映射归一到最新 taxonomy，保证跨批次、跨版本统计仍准确。

### 2.3 大批量分析机制

聚合分析脚本，扫描全部 Case 文件：

- 解析 `[meta]` 头部（tags / difficulty / variant / source / date），按 taxonomy aliases 归一到最新版本。
- 输出崩坏率的交叉统计：`崩坏分类（父/子） × 难度 × 变体 × 时间/来源`，及高频模式排行。
- Case 量增长后靠结构化 tags 聚合保持准确；自由文本 feedback 仅供人工下钻，不参与自动统计。
- 首版直接服务 exp002 揭盲（control vs rulepack 分 tags 崩坏率对比）。

## 3. 腿脚/袜子描写规范（规则包，落点为生产模板）

六条规则，作用于 step2 good_template 腿脚相关段落与 step1 姿势决策。**slim 六条瘦身规则不动**，本规则包是其上的腿脚专项增量。

- **R1 材质词黑白名单**：袜子/腿脚段禁用光学-材料学词汇（折射、珠光、透光率、微孔织物、丝线密度、次表面散射、莹润等），改用感性白名单（轻盈、柔和光泽、服帖、朦胧）。治塑料袋感 + 画风写实化（假设 H1，兼治 Case02 型画风崩坏）。
- **R2 袜口结构锚定必选**：凡出现袜子，必须写一个结构特征（蕾丝袜口 / 罗纹袜口 / 缎带等），给模型"这是一件织物"的锚。治上色感（H2）。
- **R3 脚趾防护标配**：正文固定"足尖在袜面包裹下轮廓圆润柔和"类写法；Negative 标配"裸露脚趾、脚趾根根分明"。治脚趾崩（H3）。
- **R4 褶皱单处克制**：至多一处褶皱描述，禁用"细密 / 明显 / 夸张厚度"等程度词。治皱褶夸张（H4）。
- **R5 高难姿势退让**：step1 决策时规避"深度折叠 / 肢体多重交叠"极端姿势，替换为同场景舒展变体。**腿脚展示量不减**，牺牲的只是姿势极端性——不违反腿脚自然展示第一原则（H5）。
- **R6 Negative 腿脚位保底**：Negative 5 条限额内，腿脚相关条目（多肢 / 脚趾 / 袜子缺失类）保底占位，优先级高于泛用条目。

具体措辞在 writing-plans 阶段结合 good_template 现状细化；本节只定规则意图。

### 根因假设映射（供实验归因，非结论）

| 假设 | 现象 | 规则 |
|------|------|------|
| H1 材质描述过度物理化把渲染推向写实 | bad Case05 塑料袋感 3/3、Case02 画风崩成写实 3D 3/3；good 材质描述更短更感性 | R1 |
| H2 袜子缺结构锚定→被画成一层色 | good 两例袜口有蕾丝；bad 各例无结构 | R2 |
| H3 脚趾无防护 | good Case01 有防护、bad Case03 脚趾根根分明 | R3 |
| H4 褶皱描写过密带程度词→夸张呈现 | bad Case01/03/04 皱褶夸张 | R4 |
| H5 高难姿势本身是比例崩坏高发区 | bad Case01 抱膝深折比例不协调；exp001 结论一致 | R5 |

## 4. exp002 实验设计

- **对照**：control = 当前生产链路原样产出；treatment = 规则包版链路产出。同种子、同参考图；画幅跟随各自决策块（runner `resolve_send_inputs` 已具备）。
- **基准集 benchmark_v2（腿脚高危侧重）**：保留 v1 的 medium/hard 种子（盘腿、鸭子坐、高背椅伸展）；从本批 bad cases 补入丝袜高危场景（白丝跪坐卧室、薄纱短袜地板坐、抱膝床铺）。约 8–10 种子、两角色（castorice / hysilens）。种子文本 Claude 起草、用户审定后冻结。
- **规模**：每格 5 张 → 2 变体 × 9 种子 × 5 张 ≈ 90 张/轮（用户批准"可以更大"；exp001 为 36）。
- **盲评**：沿用揭盲页机制（`_reveal.py`）。盲评表 = 2.1 的 feedback 待填模板，评完即归档为 Case（source=exp002），不做两遍。tags 由用户填或 Claude 依 feedback 代填后用户确认。
- **判定**：
  - 主指标：腿脚崩坏率（父 tag 脚部/腿部/袜子，分子 tag 统计），期望相对 ~22% 基线显著下降。
  - 护栏指标：脸部还原、构图、画风不回退（画风崩坏率纳入统计）。
  - 判定框架沿用 exp001：区分"模板遵从性问题"（LLM 没照做 → 迭代模板措辞）与"规则本身问题"（自动链路下有副作用 → 回退该条规则）。

## 5. 硬约束（贯穿全阶段）

1. **腿脚自然展示第一原则不可违反**：任何规则不得靠"藏脚 / 减少腿脚出镜"降崩坏率。R5 只限制极端姿势，不限制腿脚展示本身；牺牲姿势多样性可以，牺牲核心价值点不行。
2. **slim 基准不动摇**：规则包是 slim 之上的腿脚专项增量，不推翻 exp001 六条瘦身规则。
3. **冻结件不可编辑**：`experiments/variants/exp001/**` 等已冻结实验件永不回改。
4. **config.py 保密**：`app/tools/llm/config.py` gitignored，含 API key，不入库不外发。

## 6. B/C 辅助路线（写入备选，本阶段不实施）

- **B 单规则消融**：exp002 整体有效但某类崩坏无改善 → 对该类对应规则做小规模消融归因（每因子独立变体）。首轮不做，因腿脚崩坏是概率事件，逐因子首轮样本量不足反而测不准。
- **C 链路手段**：Prompt 层到平台期（连续两轮迭代崩坏率无显著下降且仍高于可接受线，可接受线由用户届时定）→ 评估崩坏检测自动重 roll、腿脚特写参考图注入、生成参数调整。

## 7. 组件边界（供 writing-plans 分解）

- `experiments/cases/taxonomy.yaml`：崩坏分类活字典（version + 两层 tag + aliases）。
- Case 归档模板生成器：出图产物 → 预填 [meta] + 空 feed_back 的 Case 文件（feedback 待填）。
- 聚合分析脚本：扫描 Case → taxonomy 归一 → 交叉统计 + 高频排行。
- `experiments/configs/exp002.yaml` + `benchmark_v2.yaml`：exp002 实验配置与高危基准集。
- 规则包落点：step2 good_template 腿脚段 + step1 姿势决策措辞（具体文件在 plan 阶段定位）。

每个组件单一职责、接口清晰、可独立测试。规则包对生产模板的改动须可被 exp002 A/B 验证后再合入，遵循 exp001/exp001b 已验证的"实验先行、揭盲定论"流程。
